Flujo docente completo con Quarto

Del repositorio vacío al curso publicado, de punta a punta

Los capítulos anteriores cubrieron las piezas: el sitio web, las slides, los documentos reproducibles, la publicación. Este capítulo las ensambla en un flujo de trabajo completo — desde el primer git init hasta el sitio del curso publicado con material de todas las semanas, integrado con Colab, con LaTeX donde es necesario, y con un protocolo claro para mantenerlo semestre a semestre.

No es un capítulo teórico. Es una guía de acción.

El repositorio como sitio del curso

El cambio de paradigma

La mayoría de los cursos universitarios tiene el material disperso en múltiples lugares: el programa en el sistema académico, los apuntes en Google Drive, las slides en la nube, los datos en un pendrive, las soluciones en el correo. Cada semestre se repite el mismo caos de buscar “¿dónde puse la versión final del lab 3?”.

El repositorio como sitio del curso resuelve esto con un principio simple: un solo lugar de verdad.

Todo en un lugar. Con historial. Colaborativo. Automático.

El modelo de dos capas

El repositorio tiene exactamente dos tipos de contenido:

Capa 1 — Código fuente (branch main): Todo lo que tú editas: archivos .qmd, datos, SCSS, configuración YAML, notebooks .ipynb.

Capa 2 — Sitio publicado (automático via GitHub Actions): El HTML, CSS y JS generados. Los estudiantes acceden aquí. Tú nunca lo editas directamente.

main (tú editas aquí)          gh-pages (generado automáticamente)
├── index.qmd          →       ├── index.html
├── lectures/          →       ├── lectures/
│   ├── semana01/      →       │   ├── semana01/
│   │   └── index.qmd  →       │   │   └── index.html

Tú trabajas solo en main. El sitio se actualiza solo.

Syllabus interactivo en Quarto

Por qué un syllabus como página web

Un syllabus en PDF es estático — está desactualizado en el momento en que cambias una fecha. Un syllabus en Quarto puede tener:

  • Fechas que calculan automáticamente los días restantes.
  • Links directos al material de cada semana.
  • Estado de cada evaluación (pendiente/abierta/cerrada).
  • Calendario embebido o descargable.
  • Búsqueda integrada en el sitio.

Política de uso de IA

TipUso permitido con declaración

En este curso se permite el uso de herramientas de IA generativa (ChatGPT, Claude, Copilot, etc.) con las siguientes condiciones:

Permitido sin restricción: - Depurar errores de código - Entender mensajes de error - Buscar documentación de funciones - Corregir gramática en informes

Permitido con declaración explícita: - Asistencia en la escritura de código (declara qué partes) - Generación de borradores de texto (declara y revisa)

No permitido: - Entregar código o texto generado íntegramente por IA sin revisión - No declarar el uso cuando lo hay - Usar IA en evaluaciones presenciales

Si tienes dudas sobre un caso específico, consulta antes de hacerlo.

### El problema de las soluciones

Las soluciones de laboratorios y tareas no pueden vivir en el mismo branch público que el enunciado. Tienes tres opciones:

**Opción A — Branch privado de soluciones:**

```bash
# Crear y mantener branch solo local / en repo privado
git checkout -b soluciones
# Agregar soluciones aquí
git add labs/lab03/solucion/solucion.qmd
git commit -m "feat: solución lab 3"
# NUNCA hacer push de este branch al repo público

Opción B — Repositorio privado separado:

mat281-2025-1/        (público — estudiantes acceden)
mat281-2025-1-private/ (privado — solo docente y ayudantes)
    └── soluciones/
        ├── lab01/solucion.qmd
        ├── lab02/solucion.qmd
        └── ...

Opción C — Carpeta excluida con .gitignore:

# En .gitignore del repo público
labs/*/solucion/
assignments/*/solucion/
examenes/*/pauta/

Las soluciones existen localmente pero nunca se suben a GitHub.

TipLa opción más práctica para docentes: Opción C

Crea la carpeta solucion/ dentro de cada lab, ponla en .gitignore, y trabaja normalmente. El enunciado se publica automáticamente; la solución queda solo en tu máquina. Comparte la solución por el LMS o Teams cuando sea el momento.

Sincronizar el .qmd con el .ipynb

Puedes trabajar en Quarto (.qmd) y exportar el notebook para Colab automáticamente:

# Convertir .qmd a .ipynb para Colab
quarto convert lectures/semana03/index.qmd

# O renderizar directamente a notebook
quarto render lectures/semana03/index.qmd --to ipynb

Esto genera index.ipynb que los estudiantes pueden abrir en Colab directamente desde GitHub.

GitHub Action para sincronizar automáticamente

# .github/workflows/sync-notebooks.yml
name: Sincronizar notebooks para Colab

on:
  push:
    branches: [main]
    paths:
      - "lectures/**/*.qmd"
      - "labs/**/*.qmd"

jobs:
  sync:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: quarto-dev/quarto-actions/setup@v2
        with:
          version: "1.5.57"

      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"
          cache: "pip"

      - run: pip install -r requirements.txt

      - name: Convertir QMDs a IPYNBs
        run: |
          for qmd in lectures/semana*/index.qmd labs/lab*/enunciado.qmd; do
            echo "Convirtiendo: $qmd"
            quarto convert "$qmd" --output "${qmd%.qmd}.ipynb"
          done

      - name: Commit y push de notebooks
        run: |
          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          git add lectures/**/*.ipynb labs/**/*.ipynb
          git diff --staged --quiet || git commit -m "chore: actualizar notebooks para Colab"
          git push

Combinar Quarto con LaTeX para ecuaciones avanzadas

Ecuaciones en Quarto: LaTeX nativo

Quarto usa MathJax (para HTML) y el motor LaTeX (para PDF) para renderizar ecuaciones. La sintaxis es idéntica a LaTeX:

**Inline:** la media es $\bar{x} = \frac{1}{n}\sum_{i=1}^n x_i$.

**Display:**
$$
\hat{\beta} = (X^\top X)^{-1} X^\top y
$$ {#eq-mco}

**Con numeración y referencia:** ver la @eq-mco.

**Sistema de ecuaciones con `align`:**
$$
\begin{aligned}
\hat{\beta}_1 &= \frac{S_{XY}}{S_{XX}} \\
\hat{\beta}_0 &= \bar{y} - \hat{\beta}_1 \bar{x}
\end{aligned}
$$

**Matrices:**
$$
X = \begin{pmatrix}
1 & x_1 \\
1 & x_2 \\
\vdots & \vdots \\
1 & x_n
\end{pmatrix}, \quad
\beta = \begin{pmatrix} \beta_0 \\ \beta_1 \end{pmatrix}
$$

Entornos matemáticos personalizados

Para cursos con muchos teoremas y demostraciones, los entornos de Quarto:

::: {#thm-gauss-markov}
## Teorema de Gauss-Markov

Bajo los supuestos clásicos de regresión, los estimadores MCO son
los mejores estimadores lineales insesgados (MELI).
:::

::: {#prp-invarianza-mco}
## Invarianza ante transformaciones lineales

Si $\tilde{X} = aX + b$ con $a \neq 0$, entonces
$\hat{\beta}_1^* = \hat{\beta}_1 / a$ y $\hat{\beta}_0^* = \hat{\beta}_0 - b\hat{\beta}_1/a$.
:::

::: {.proof}
Sea $\tilde{S}_{XY} = \sum(\tilde{x}_i - \bar{\tilde{x}})(y_i - \bar{y})$...
:::

El @thm-gauss-markov justifica el uso de MCO bajo los supuestos estándar.

Ahora en cualquier .qmd del curso:

Sea $X_i \sim \N(\mu, \sigma^2)$. Entonces $\E{X_i} = \mu$ y $\Var{X_i} = \sigma^2$.

Si $X \ind Y$, entonces $\E{XY} = \E{X}\E{Y}$.

Script de verificación de links

# scripts/check_links.py
# Verifica que todos los links externos del sitio funcionen

import re
import requests
from pathlib import Path
from concurrent.futures import ThreadPoolExecutor

def extract_links(qmd_file):
    """Extrae todos los URLs de un archivo .qmd."""
    content = Path(qmd_file).read_text()
    return re.findall(r'https?://[^\s\)\]"]+', content)

def check_link(url):
    """Verifica si un URL responde correctamente."""
    try:
        r = requests.head(url, timeout=10, allow_redirects=True)
        return url, r.status_code, r.status_code < 400
    except Exception as e:
        return url, 0, False

def main():
    # Encontrar todos los .qmd
    qmds = list(Path('.').rglob('*.qmd'))
    qmds = [f for f in qmds if '_site' not in str(f) and '_freeze' not in str(f)]

    # Extraer todos los links
    all_links = set()
    for qmd in qmds:
        all_links.update(extract_links(qmd))

    # Filtrar links internos y de assets
    external = [l for l in all_links
                if not any(s in l for s in ['github.com/fralfaro', 'localhost',
                                             'colab.research.google.com/github/fralfaro'])]

    print(f"Verificando {len(external)} links externos...")

    broken = []
    with ThreadPoolExecutor(max_workers=10) as ex:
        results = list(ex.map(check_link, external))

    for url, status, ok in results:
        if not ok:
            broken.append((url, status))
            print(f"  ❌ {status} → {url}")
        else:
            print(f"  ✅ {status} → {url[:60]}")

    if broken:
        print(f"\n⚠️  {len(broken)} links rotos. Corregir antes de publicar.")
    else:
        print(f"\n✅ Todos los links funcionan.")

if __name__ == '__main__':
    main()
# Ejecutar al inicio de cada semestre
pip install requests
python scripts/check_links.py
