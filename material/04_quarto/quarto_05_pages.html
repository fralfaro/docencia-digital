Publicación con GitHub Pages

De archivo local a sitio web público en minutos

El material de un curso que vive solo en tu computador existe para ti. El material que vive en un sitio web público existe para todos — tus estudiantes de este semestre, los del próximo, colegas de otras universidades que buscan recursos, y tú mismo cuando necesitas el apunte en un dispositivo que no es el tuyo.

GitHub Pages convierte tu repositorio Quarto en un sitio web sin costo, sin configuración de servidor, y con actualizaciones automáticas cada vez que haces push. Este capítulo cubre todas las formas de publicar — desde el comando más rápido hasta el pipeline de CI/CD más robusto — y cierra con el caso de un repositorio de charlas públicas publicado como portafolio.

quarto publish gh-pages: publicación directa

El comando más rápido

Si tienes el repositorio configurado y Quarto instalado, publicar es un solo comando:

quarto publish gh-pages

Quarto hace todo automáticamente:

  1. Renderiza el proyecto completo.
  2. Crea (o actualiza) el branch gh-pages con el HTML generado.
  3. Hace push del branch a GitHub.
  4. GitHub Pages sirve ese branch como sitio web.

El sitio queda disponible en:

https://tu-username.github.io/nombre-repositorio

Primera publicación paso a paso

# 1. Asegúrate de estar en el directorio raíz del proyecto
cd mat281-2025-1

# 2. Asegúrate de que el repositorio tiene remote configurado
git remote -v
# origin  https://github.com/tu-username/mat281-2025-1.git (fetch)
# origin  https://github.com/tu-username/mat281-2025-1.git (push)

# 3. Publicar
quarto publish gh-pages

# Quarto pregunta:
# ? Would you like to create a new site? (Y/n) Y
# → Crea el branch gh-pages y publica

# En publicaciones siguientes, simplemente:
quarto publish gh-pages

Activar GitHub Pages en el repositorio

La primera vez que publicas con quarto publish gh-pages, puede ser necesario activar Pages manualmente:

  1. Ve al repositorio en GitHub.
  2. Settings → Pages.
  3. Source: Deploy from a branch.
  4. Branch: gh-pages / Folder: / (root).
  5. Clic Save.

En 1-2 minutos el sitio está disponible.

Variantes del comando

# Publicar sin render previo (usa el _site/ ya generado)
quarto publish gh-pages --no-render

# Publicar sin confirmación interactiva (útil en scripts)
quarto publish gh-pages --no-prompt

# Publicar a un repositorio diferente (no el del remote "origin")
quarto publish gh-pages --remote origin

# Verificar qué se publicaría sin hacer nada
quarto publish gh-pages --dry-run

# Publicar a Quarto Pub (alternativa gratuita a GitHub Pages)
quarto publish quarto-pub

# Publicar a Netlify
quarto publish netlify

# Publicar a Posit Connect
quarto publish connect

El branch gh-pages: qué contiene

# Ver el contenido del branch gh-pages sin cambiarte a él
git show gh-pages:index.html | head -20

# Listar todos los archivos en gh-pages
git ls-tree --name-only gh-pages

# El branch contiene solo los archivos renderizados:
# index.html
# lectures/semana01/index.html
# lectures/semana01/slides.html
# assets/css/...
# assets/js/...
# ...

El branch gh-pages no debe modificarse manualmente — quarto publish lo reescribe completo en cada publicación. El código fuente (.qmd) vive solo en main.

Tip.nojekyll es obligatorio

Quarto agrega automáticamente un archivo .nojekyll en el branch gh-pages. Este archivo le dice a GitHub que no procese el sitio con Jekyll (el generador de sitios por defecto de GitHub Pages). Sin él, los paths de CSS y JS se rompen porque Jekyll ignora carpetas que empiezan con _ (como _site/).

Si algo falla y reconstruyes el branch manualmente, recuerda incluirlo:

touch .nojekyll

GitHub Actions para Quarto: workflow automático

Por qué usar Actions en lugar de quarto publish

quarto publish gh-pages es perfecto para publicar desde tu máquina. Pero tiene limitaciones:

  • Requiere que estés en tu computador para publicar.
  • Si trabajas con ayudantes que hacen push, ellos no pueden publicar.
  • Si el código tarda minutos en ejecutarse, bloquea tu terminal.

GitHub Actions resuelve todo esto: el deploy ocurre en la nube, automáticamente, cada vez que alguien hace push a main — sin importar quién ni desde dónde.

El workflow completo y comentado

# .github/workflows/deploy.yml
name: Deploy Quarto a GitHub Pages

on:
  push:
    branches: [main]           # dispara en cada push a main
  workflow_dispatch:           # permite ejecución manual desde la UI

# Permisos necesarios para escribir en GitHub Pages
permissions:
  contents: read
  pages: write
  id-token: write

# Evitar deploys concurrentes — si ya hay uno corriendo, espera
concurrency:
  group: "pages"
  cancel-in-progress: false    # no cancela el deploy en curso

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      # ── 1. Obtener el código fuente ─────────────────────────
      - name: Checkout del repositorio
        uses: actions/checkout@v4
        with:
          fetch-depth: 0       # historial completo (necesario para last-modified)

      # ── 2. Restaurar cache de Quarto (freeze) ───────────────
      - name: Restaurar cache de Quarto freeze
        uses: actions/cache@v4
        with:
          path: _freeze
          key: quarto-freeze-${{ hashFiles('**/*.qmd', '**/*.ipynb') }}
          restore-keys: |
            quarto-freeze-

      # ── 3. Instalar Quarto ──────────────────────────────────
      - name: Instalar Quarto
        uses: quarto-dev/quarto-actions/setup@v2
        with:
          version: "1.5.57"    # fijar versión para reproducibilidad

      # ── 4. Configurar Python ────────────────────────────────
      - name: Configurar Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.11"
          cache: "pip"         # cachea ~/.cache/pip entre runs

      # ── 5. Instalar dependencias Python ─────────────────────
      - name: Instalar dependencias
        run: |
          pip install --upgrade pip
          pip install -r requirements.txt

      # ── 6. (Opcional) Configurar R ──────────────────────────
      # Solo si el proyecto tiene chunks de R
      # - name: Configurar R
      #   uses: r-lib/actions/setup-r@v2
      #   with:
      #     r-version: "4.3.0"
      #
      # - name: Instalar paquetes R
      #   uses: r-lib/actions/setup-r-dependencies@v2

      # ── 7. Configurar GitHub Pages ──────────────────────────
      - name: Configurar GitHub Pages
        id: pages
        uses: actions/configure-pages@v4

      # ── 8. Renderizar el proyecto Quarto ────────────────────
      - name: Renderizar Quarto
        uses: quarto-dev/quarto-actions/render@v2
        with:
          to: html               # formato de salida
          # path: .              # raíz del proyecto (por defecto)

      # ── 9. Subir el sitio como artefacto para Pages ─────────
      - name: Subir artefacto
        uses: actions/upload-pages-artifact@v3
        with:
          path: "_site"          # carpeta generada por quarto render

  deploy:
    needs: build                 # espera a que build termine exitosamente
    runs-on: ubuntu-latest

    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}

    steps:
      - name: Desplegar en GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4

Activar GitHub Pages con Actions como fuente

Repositorio → Settings → Pages → Source: GitHub Actions

Esto le dice a GitHub que el deploy lo maneja Actions, no el branch gh-pages.

Ajustar _quarto.yml para el sitio desplegado

website:
  site-url: "https://tu-username.github.io/mat281-2025-1"

  # Ruta base si el sitio no está en la raíz del dominio
  # (necesario cuando el repo no es "tu-username.github.io")
  base-url: "/mat281-2025-1"
Advertenciasite-url es importante para Open Graph y sitemap

Sin site-url, los metadatos de Open Graph (las tarjetas al compartir en redes sociales) y el sitemap.xml generado no tendrán URLs correctas. Siempre configúralo con la URL final del sitio.

Cache de dependencias para deploys rápidos

El cache de pip y de _freeze son los dos optimizaciones más importantes. Sin ellas, cada deploy re-instala todas las librerías y re-ejecuta todo el código:

# Cache de pip — entre 30s y 3min de ahorro por run
- uses: actions/setup-python@v5
  with:
    python-version: "3.11"
    cache: "pip"               # cachea ~/.cache/pip

# Cache de Quarto freeze — ahorro de minutos si hay código pesado
- uses: actions/cache@v4
  with:
    path: _freeze
    key: quarto-freeze-${{ hashFiles('**/*.qmd') }}

Tiempos típicos de deploy con cache activo:

Tipo de sitio Sin cache Con cache
Solo Markdown (sin código) 2-3 min 1-2 min
Con código Python (freeze) 10-20 min 2-4 min
Con código Python y R 15-30 min 3-6 min

Deploy en cada push al branch principal

Proteger main para que el deploy sea confiable

Si cualquier push a main dispara el deploy, un commit con error puede publicar un sitio roto. La solución es requerir que los cambios pasen por un PR y que el CI los valide antes de mergear:

# .github/workflows/check.yml — workflow de validación (en PRs)
name: Verificar render

on:
  pull_request:
    branches: [main]

jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: quarto-dev/quarto-actions/setup@v2
        with:
          version: "1.5.57"
      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"
          cache: "pip"
      - run: pip install -r requirements.txt

      - name: Verificar render sin deploy
        uses: quarto-dev/quarto-actions/render@v2
        with:
          to: html

Con esto el flujo es:

Tu editas en un branch → abres PR → CI verifica que renderiza →
apruebas el PR → merge a main → deploy automático

Notificaciones de deploy

# Al final del job deploy:
      - name: Notificar resultado
        if: always()
        run: |
          if [ "${{ steps.deployment.outcome }}" == "success" ]; then
            echo "✅ Deploy exitoso: ${{ steps.deployment.outputs.page_url }}"
          else
            echo "❌ Deploy falló — revisar logs"
          fi

Para recibir notificaciones por email, GitHub las envía automáticamente cuando un workflow falla en main.

Deploy condicional: solo si cambiaron archivos .qmd

on:
  push:
    branches: [main]
    paths:
      - "**.qmd"           # solo si cambió algún .qmd
      - "**.ipynb"         # o algún notebook
      - "_quarto.yml"      # o la configuración
      - "**.scss"          # o los estilos
      - "requirements.txt" # o las dependencias

Esto evita deploys innecesarios cuando solo cambia el README.md o los datos.

Múltiples entornos: preview y producción

Para proyectos más grandes, puedes tener un sitio de preview (staging) y uno de producción:

on:
  push:
    branches: [main]       # → deploy a producción
  push:
    branches: [dev]        # → deploy a preview (rama de desarrollo)

jobs:
  deploy-prod:
    if: github.ref == 'refs/heads/main'
    # ... deploy a github-pages

  deploy-preview:
    if: github.ref == 'refs/heads/dev'
    # ... deploy a Netlify u otro servicio

Configurar dominio personalizado

Opciones de dominio para sitios docentes

Opción URL resultante Costo Configuración
GitHub Pages por defecto username.github.io/repo Gratis Automático
Subdominio del dominio institucional mat281.usm.cl Gratis (lo gestiona TI) Contactar TI de UTFSM
Dominio propio mat281.juanperez.cl ~USD 10/año DNS + CNAME
Subdominio de dominio propio cursos.juanperez.cl Incluido en el dominio DNS + CNAME

Configurar un dominio personalizado en GitHub Pages

Paso 1 — Agregar el archivo CNAME al repositorio:

# En la raíz del proyecto (branch main)
echo "mat281.juanperez.cl" > CNAME
git add CNAME
git commit -m "chore: agregar dominio personalizado"
git push

O configúralo en _quarto.yml para que Quarto lo incluya automáticamente:

project:
  resources:
    - CNAME

Con el contenido del archivo CNAME:

mat281.juanperez.cl

Paso 2 — Configurar el DNS de tu dominio:

En el panel de tu registrador de dominio (GoDaddy, Namecheap, Cloudflare, etc.), agrega un registro CNAME:

Tipo:   CNAME
Nombre: mat281
Valor:  tu-username.github.io.   ← (con el punto final)
TTL:    3600

O si quieres usar el dominio raíz (juanperez.cl sin subdominio), agrega registros A:

Tipo:   A
Nombre: @
Valor:  185.199.108.153
        185.199.109.153
        185.199.110.153
        185.199.111.153

Paso 3 — Configurar en GitHub:

  1. Repositorio → Settings → Pages.
  2. Custom domain: escribe mat281.juanperez.cl.
  3. Marca Enforce HTTPS (requiere que el DNS esté propagado).

La propagación DNS tarda entre 5 minutos y 48 horas según el registrador.

Paso 4 — Actualizar site-url en _quarto.yml:

website:
  site-url: "https://mat281.juanperez.cl"

Dominio institucional de la UTFSM

Para obtener un subdominio bajo usm.cl (por ejemplo mat281.usm.cl):

  1. Contacta a la Dirección de Informática / DTD de la UTFSM.
  2. Solicita un registro CNAME apuntando a tu-username.github.io.
  3. Ellos configuran el DNS institucional.
  4. Tú configuras el dominio en GitHub Pages y actualizas site-url.

Gestión de versiones y registro de cambios

Versionar el sitio por semestre con Git tags

# Al finalizar el semestre, etiquetar la versión estable
git tag -a v2025-1 -m "Cierre semestre 2025-1 — material completo"
git push origin v2025-1

# Ver todos los tags
git tag -l

# Restaurar el estado del repositorio en ese tag (solo lectura)
git checkout v2025-1

# Volver al presente
git checkout main

Los tags funcionan como “puntos de guardado” — puedes volver exactamente a cómo estaba el curso en el semestre 2024-2, por ejemplo.

**Prioridad:** Media.

Caso de cierre: repositorio de charlas y talks publicado

El contexto

Francisco Alfaro tiene un repositorio de charlas públicas en github.com/fralfaro/talks con más de 40 presentaciones de eventos como PyCon Chile, PyCon Colombia, useR!, LatinR, Nerdearla y otros. ### GitHub Actions para el repositorio de talks

```yaml
# .github/workflows/deploy.yml
name: Deploy talks a GitHub Pages

on:
  push:
    branches: [main]
    paths:
      - "talks/**"
      - "index.qmd"
      - "_quarto.yml"
      - "**.scss"
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: Instalar Quarto
        uses: quarto-dev/quarto-actions/setup@v2
        with:
          version: "1.5.57"

      # Las slides son Markdown puro — no necesitamos Python
      # Solo Quarto para renderizar

      - uses: actions/configure-pages@v4

      - name: Renderizar
        uses: quarto-dev/quarto-actions/render@v2
        with:
          to: html

      - uses: actions/upload-pages-artifact@v3
        with:
          path: "_site"

  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - uses: actions/deploy-pages@v4
        id: deployment

Como las slides son Markdown puro (sin código Python ejecutado), el deploy tarda solo 1-2 minutos.

# CLAUDE.md — Repositorio de charlas de fralfaro

## Contexto
Este repositorio contiene charlas técnicas de Francisco Alfaro
(fralfaro), Jefe de Análisis y Modelación en UTFSM. Las charlas
cubren ciencia de datos, Python, educación digital y herramientas
de IA aplicada.

## Estructura de una charla nueva
Para agregar una charla nueva, crea:
1. `talks/EVENTO-AÑO/index.qmd` — descripción y metadata
2. `talks/EVENTO-AÑO/slides.qmd` — presentación RevealJS
3. `talks/EVENTO-AÑO/preview.png` — imagen de portada (1200×630px)

## Frontmatter requerido para index.qmd
```yaml
---
title: "Título de la charla"
date: "YYYY-MM-DD"
description: "Descripción de 1-2 oraciones."
categories: [tema1, tema2, tema3]
image: "preview.png"
event: "Nombre del evento"
event-url: "https://..."
location: "Ciudad, País"
---

Estilo de las slides

  • Tema: [default, ../../assets/scss/slides.scss]
  • Logo: ../../assets/img/brand/fralfaro.png
  • Footer: "fralfaro.github.io/talks"
  • Duración típica: 25-30 slides para charla de 30 min
  • Incluir siempre: slide de agenda, slide de conclusiones, slide de contacto al final ```

Resultado final del sitio de talks

El sitio publicado en fralfaro.github.io/talks tiene:

  • Página de inicio con el grid de charlas organizadas por año
  • Filtro por categorías (python, quarto, educación, IA, etc.)
  • Cada charla con descripción, slides embebidas, links al evento y al código
  • Búsqueda que indexa títulos y descripciones
  • Modo oscuro sincronizado con el sistema operativo
  • Deploy automático en ~1 minuto en cada push
