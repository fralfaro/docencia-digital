El argumento para publicar

“El material de un curso que vive solo en tu computador existe para ti.

El material que vive en un sitio web público existe para todos.”

  • Tus estudiantes de este semestre
  • Los del próximo semestre
  • Colegas de otras universidades que buscan recursos
  • Tú mismo cuando necesitas el apunte en un dispositivo que no es el tuyo

GitHub Pages

  • Gratis (para repos públicos)
  • Sin configuración de servidor
  • Dominio automático: usuario.github.io/repo
  • Actualizaciones automáticas con cada push

quarto publish gh-pages: el comando más rápido

Si tienes el repositorio configurado:

# Un solo comando
quarto publish gh-pages

Quarto hace automáticamente:

  1. Renderiza el sitio localmente
  2. Crea (o actualiza) el branch gh-pages
  3. Sube los HTML al branch
  4. GitHub Pages sirve el contenido

Primera vez, pregunta:

? Publish site to:
  ❯ github-pages
? GitHub Repository: usuario/mat281-2025-1
? Publish to: https://usuario.github.io/mat281-2025-1

Condiciones previas:

# Verificar que estás en un repo de GitHub
git remote -v

# Quarto instalado
quarto --version

# gh CLI autenticada (opcional pero recomendado)
gh auth status

Para publicar el sitio del curso

Este es el método más rápido para empezar. Luego puedes migrar a GitHub Actions cuando quieras automatizar más.

Las tres rutas de publicación

Opción A: quarto publish

  • Render local + push automático
  • Sin configuración de GitHub Actions
  • Ideal para empezar rápido
  • Requiere correr el comando manualmente cada vez
quarto publish gh-pages

Opción B: /docs en main

  • Render local → commitear docs/
  • GitHub Pages sirve desde main/docs
  • Más control, sin Actions
# _quarto.yml
project:
  output-dir: docs
quarto render
git add docs/
git commit -m "render"
git push

Opción C: GitHub Actions

  • Render automático en cada push
  • Sin archivos generados en el repo
  • El método más robusto
# .github/workflows/deploy.yml
on:
  push:
    branches: [main]

Activar GitHub Pages en el repositorio

Después de correr quarto publish (o configurar Actions):

  1. Repositorio en GitHub
  2. Settings → Pages
  3. Source:
    • Si usas quarto publish: Deploy from branch → gh-pages → /root
    • Si usas docs/: Deploy from branch → main → /docs
    • Si usas Actions: GitHub Actions
  4. Clic en Save

En 1-2 minutos el sitio está en:

https://usuario.github.io/mat281-2025-1

Verificar que funciona:

  • Tab Pages en Settings → URL del sitio activo
  • Tab Actions → último deploy exitoso ✅
  • Abrir la URL en el navegador

404 al principio

Si el sitio muestra 404 inmediatamente después de activar, espera 1-2 minutos. GitHub Pages tarda un poco en propagar la primera vez.

El workflow de Actions para GitHub Pages

# .github/workflows/deploy-pages.yml
name: Deploy Quarto a GitHub Pages

on:
  push:
    branches: [main]
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: quarto-dev/quarto-actions/setup@v2
        with:
          version: '1.5.57'
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
          cache: 'pip'
      - run: pip install -r requirements.txt
      - uses: actions/configure-pages@v4
      - uses: quarto-dev/quarto-actions/render@v2
      - uses: actions/upload-pages-artifact@v3
        with:
          path: '_site'

  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - uses: actions/deploy-pages@v4

_freeze/: el secreto del deploy rápido

Sin freeze:

Cada deploy ejecuta todo el código de todos los notebooks. Con 20 notebooks de 5 minutos cada uno → 100 minutos de build.

Con freeze: auto:

# _quarto.yml
execute:
  freeze: auto

Quarto guarda los resultados en _freeze/. Si el código no cambió, no lo re-ejecuta.

Build de 100 min → 3-5 min.

Implementación:

# Después de renderizar localmente
git add _freeze/
git commit -m "chore: actualizar freeze cache"
git push

En el workflow de Actions, _freeze/ ya está en el repo. Solo re-ejecuta lo que cambió.

Commitear _freeze/

Esto es inusual — normalmente los archivos generados no van al repo. Con Quarto, _freeze/ sí debe estar en el repo para que el deploy automático sea rápido.

Dominio personalizado (opcional)

Si quieres una URL como mat281.dmat.usm.cl en lugar de fralfaro.github.io/mat281-2025-1:

1. Agregar archivo CNAME en la raíz:

mat281.dmat.usm.cl

2. En GitHub Pages Settings:

Settings → Pages → Custom domain → mat281.dmat.usm.cl

3. Configurar DNS en la institución:

CNAME record: mat281.dmat.usm.cl → fralfaro.github.io

(El administrador de TI de la universidad configura esto)

HTTPS automático

GitHub Pages activa automáticamente HTTPS (certificado Let’s Encrypt) para dominios personalizados. No requiere configuración adicional.

Para proyectos Quarto, agregar en _quarto.yml:

website:
  site-url: "https://mat281.dmat.usm.cl"

Esto asegura que las URLs generadas sean correctas para el dominio personalizado.
