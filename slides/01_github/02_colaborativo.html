El problema que resuelve el PR

Escenario típico:

Tu ayudante quiere agregar la solución del laboratorio 3 al repositorio del curso.

  • Si le das acceso directo → puede hacer push de cualquier cosa al main
  • Si no le das acceso → no puede contribuir

El mecanismo Fork + Pull Request resuelve esto:

El ayudante trabaja en su propia copia, hace los cambios, y propone integrarlos. Tú revisas y decides.

Fork: una copia independiente

El ayudante:

  1. Va a github.com/tu-username/mat281
  2. Clic en Fork → selecciona su cuenta
  3. GitHub crea github.com/ayudante/mat281
# Clona su fork (no el original)
git clone git@github.com:ayudante/mat281.git
cd mat281

# Agrega referencia al repo original
git remote add upstream git@github.com:tu-username/mat281.git

origin vs upstream

origin = el fork del ayudante (donde hace push)

upstream = el repositorio original (de donde sincroniza)

Sincronizar el fork cuando hay cambios nuevos:

git fetch upstream
git switch main
git merge upstream/main
git push origin main

Crear y revisar un Pull Request

El ayudante:

# Crear branch dedicado
git checkout -b lab03-solucion

# Trabaja... agrega labs/lab03/solucion.ipynb

git add labs/lab03/solucion.ipynb
git commit -m "feat: solución lab 3 con comentarios"
git push origin lab03-solucion

→ GitHub muestra banner: “Compare & pull request”

En el PR escribe: - ¿Qué hace? - ¿Cómo verificar que funciona? - ¿Hay algo pendiente?

Tú como revisor:

  • Pestaña Files changed → diff línea a línea
  • Clic en + para comentar una línea específica
  • Tipos de comentario:
    • 💬 Pregunta
    • ✏️ Sugerencia
    • 🐛 Error
    • ✅ Aprobación

GitHub permite sugerir cambios

El ayudante los acepta con un clic → commit automático.

Opciones de merge

Opción Qué hace Cuándo usarla
Create a merge commit Une los dos historiales Por defecto, conserva contexto
Squash and merge Aplasta todos los commits del PR en uno PR con muchos commits pequeños de WIP
Rebase and merge Replay lineal sobre main Historial limpio sin commits de merge

Recomendación para cursos

Squash and merge → el PR entero como un único commit descriptivo en main. Sin ruido de “wip”, “arreglo typo”, etc.

Proteger el branch main:

Settings → Branches → Add rule → Require pull request reviews before merging ✅

Nadie (incluido tú, accidentalmente) hace push directo a main.

Resolución de conflictos

Cuándo ocurren:

Dos personas modificaron las mismas líneas del mismo archivo en branches distintos.

Cómo se ve:

<<<<<<< HEAD
modelo = LinearRegression(fit_intercept=True)
=======
modelo = LinearRegression()
coef = modelo.coef_
>>>>>>> semana03-mejoras

Resolver:

  1. Editar el archivo → dejar el código final correcto
  2. git add del archivo resuelto
  3. git commit

Prevenir conflictos:

  • Divide el trabajo por archivos, no por secciones del mismo archivo
  • Haz git pull frecuentemente antes de empezar
  • Branches cortos: 1-2 días de vida tienen menos divergencia
  • Comunica qué archivos estás modificando

Conflictos en notebooks .ipynb

Son JSON → difíciles de resolver manualmente.

Solución: nbdime

pip install nbdime
nbdime config-git --enable --global

Estrategia de branches para un curso

Convención recomendada:

main              → versión estable (lo que ven los estudiantes)
dev               → integración de trabajo en progreso
semana-XX/TEMA    → material de una semana
lab-XX/DESC       → soluciones o mejoras de un lab
fix/DESC          → corrección de error específico
semestre/2025-2   → adaptación para el próximo semestre

Branch para soluciones

Mantén soluciones en un branch soluciones que nunca se mergea a main. Los estudiantes solo ven main.

GitHub Classroom: gestionar entregas de estudiantes

El problema actual:

Sistema tradicional GitHub Classroom
Archivo comprimido por email Repositorio individual con historial
No sabes cuándo empezó git log muestra exactamente cuándo
Feedback en PDF anotado Comentarios en línea en el código
Sin control de versiones Pueden revertir errores y ver su progreso

Flujo del estudiante:

1. Clic en link de invitación
   ↓
2. GitHub crea repo privado con el template
   mat281-usm/lab03-apellido-nombre
   ↓
3. Estudiante clona y trabaja
   ↓
4. Push final antes del deadline
   ↓
5. Tú revisas y das feedback en el PR automático

Configurar GitHub Classroom paso a paso

1. Crear organización del curso:

  • GitHub → + → New organization
  • Nombre: mat281-usm
  • Plan: Free (suficiente para docencia)

2. En classroom.github.com:

  • New classroom → selecciona la org
  • Nombre: MAT281 — 2025-1

3. Crear template del lab:

  • README.md con instrucciones
  • enunciado.ipynb sin solución
  • datos/ con el dataset
  • Marcar como Template repository ✅

4. Crear assignment:

  • Título, deadline, visibilidad privada
  • Starter code → el template creado
  • Enable feedback pull requests ✅

Classroom genera un link:

https://classroom.github.com/a/XXXXX

→ Comparte con los estudiantes en el LMS.

Ver el progreso en tiempo real

Desde el panel de Classroom puedes ver todos los repositorios con su estado de actividad — incluso antes del deadline.

Templates de repositorios para reutilizar cada semestre

Un template = repositorio marcado como punto de partida. Cuando alguien crea un repo basado en él, recibe todos los archivos sin el historial de Git.

Estructura del template de curso:

plantilla-curso/
├── README.md          # con placeholders
├── LICENSE
├── .gitignore
├── _quarto.yml
├── lectures/
│   └── semana01/
├── labs/
└── datasets/
    └── raw/.gitkeep

Marcar como template: Settings → General → Template repository ✅

Usar desde CLI:

gh repo create dmat-usm/mat281-2025-2 \
  --template dmat-usm/plantilla-curso \
  --public \
  --description "MAT281, semestre 2025-2"

El valor del template

Al inicio de cada semestre: clic en “Use this template” → repositorio nuevo con la estructura lista en 30 segundos.
