¿Ya resolviste este problema… de la peor manera?

Reconoce alguno de estos:

  • apuntes_final_v3_ESTE_SI_corregido2.docx
  • Email: “ignora la versión anterior, va esta”
  • Perdiste una semana de trabajo por guardar encima del archivo equivocado
  • Tu ayudante modificó el archivo que tú también estabas editando

La solución existe

Git resuelve ese problema.

GitHub es donde ese trabajo vive, se comparte y colabora.

Git vs GitHub: la distinción que confunde a todos

Git

  • Software que corre en tu computador
  • Funciona completamente sin internet
  • Lleva el historial de cambios de tus archivos

GitHub

  • Plataforma web que almacena repos Git en la nube
  • Agrega colaboración: PRs, Issues, Pages, Actions
  • Sin Git, GitHub no existe

La analogía

Git = tu sistema de archivos local. GitHub = Google Drive diseñado para código con herramientas de colaboración.

Los tres conceptos clave: repositorio, commit, branch

Repositorio

Una carpeta que Git rastrea. Contiene tus archivos + una carpeta oculta .git/ con todo el historial.

mat281-2025-1/
├── .git/        ← historial
├── README.md
├── lectures/
└── labs/

Commit

Fotografía de todos tus archivos en un momento específico.

  • Hash único (a3f2c19)
  • Quién, cuándo, qué mensaje
  • Referencia al commit anterior

Branch

Línea paralela de desarrollo. Trabajas en una versión sin afectar la otra.

  • main → lo que ven los estudiantes
  • dev → material en preparación
  • semestre-2025-2 → siguiente semestre

El flujo de los tres estados

¿Por qué el staging?

Puedes modificar 5 archivos pero commitear solo 2. Construyes commits lógicos y coherentes aunque hayas modificado muchas cosas a la vez.

Una vez commiteado

Está guardado. Puedes volver a cualquier punto del historial en cualquier momento.

Configuración inicial (una sola vez)

git config --global user.name "Juan Pérez"
git config --global user.email "juan.perez@usm.cl"
git config --global core.editor "code --wait"   # VS Code como editor
git config --global init.defaultBranch main

Verificar:

git config --list
# user.name=Juan Pérez
# user.email=juan.perez@usm.cl
# init.defaultbranch=main

Email institucional

Usa tu @usm.cl. Es lo que aparece en el historial y lo que GitHub usa para asociar commits a tu perfil.

El ciclo de trabajo diario

# 1. Antes de empezar: traer cambios del remoto
git pull

# 2. Trabajar normalmente... editas lectures/semana03.qmd

# 3. Ver qué cambió
git status

# 4. Ver exactamente qué líneas cambiaron
git diff lectures/semana03.qmd

# 5. Agregar al staging
git add lectures/semana03.qmd

# 6. Crear el commit
git commit -m "docs: agrego material semana 3 - regresión lineal"

# 7. Subir al remoto
git push

El ritmo ideal

pull → edita → status → add → commit → push. Siempre así.

Problemas docentes resueltos con Git

Problema docente Solución con Git
“¿Cuál era la versión del enunciado que entregué?” git log muestra historial completo con fechas
“Mi ayudante modificó el archivo que yo editaba” Branches separados + merge controlado
“Quiero probar un cambio grande sin romper lo que funciona” Branch nuevo, sin tocar main
“El estudiante dice que el enunciado era diferente” git diff muestra qué cambió y cuándo
“Perdí el archivo” Está en el repositorio remoto, intacto
“Quiero reutilizar el material el próximo semestre” Tags por semestre + repositorio archivado

El README importa

Es lo primero que ven tus estudiantes, tu ayudante y cualquier colega que acceda al repositorio.

El .gitignore: qué NO rastrear

¿Por qué importa?

Sin .gitignore, Git rastrea todo — incluyendo basura que no debería estar en el repositorio.

Lo que debe ignorarse:

# Python
__pycache__/
.ipynb_checkpoints/
.env

# Quarto
_site/
_book/
_freeze/

# OS
.DS_Store
Thumbs.db

# NUNCA subir
*.key
credentials.json
token.json

Regla de oro

Si ya subiste un archivo sensible, borrarlo en un commit no es suficiente — sigue en el historial.

Revisa siempre con git status antes de git add .

Generador automático: gitignore.io

Issues y Projects: gestión docente en GitHub

Issues = unidades de trabajo pendiente

✅ Preparar guía lab 4
🐛 Error en celda 7 del notebook semana 3
💡 Agregar ejercicio de validación cruzada
📅 Actualizar dataset semana 5

Cerrar desde un commit:

git commit -m "fix: corrijo error en notebook semana 3

Closes #12"

Al hacer push, GitHub cierra el Issue #12 automáticamente.

GitHub Projects = tablero Kanban

📋 MAT281 — Planificación 2025-1

│  Backlog  │  En curso  │  Publicado  │
├───────────┼────────────┼─────────────┤
│  Lab 5    │  Semana 6  │  Semanas 1-5│
│  Lab 6    │            │  Labs 1-4   │

Planificación semestral

Crea un Issue por cada semana de material al inicio del semestre. Al final, el historial de Issues cerrados es un registro completo de todo lo preparado.

Licencias: declarar qué pueden hacer otros con tu material

Sin licencia → todos los derechos reservados por defecto, aunque el repositorio sea público.

Para código (notebooks, scripts):

Licencia Restricción
MIT Solo mantener copyright
Apache 2.0 MIT + patentes
GPL v3 Derivados también GPL

→ Recomendación: MIT

Para contenido (apuntes, slides):

Creative Commons — combina módulos:

  • BY = dar crédito al autor
  • SA = derivados con misma licencia
  • NC = solo uso no comercial
  • ND = sin derivados

→ Recomendación: CC BY 4.0

La combinación estándar en docencia

Código:    MIT License
Contenido: CC BY 4.0
